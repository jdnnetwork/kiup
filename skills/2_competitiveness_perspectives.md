# 여러분를 위한 기업 경쟁력 관점 도출

## 개요

여러분는 업계와 기업을 깊이 파고드는 사람들이다. 이 스킬은 업계 분석 자료(있을 경우)와 검색을 기반으로, 해당 기업의 경쟁력을 다각도로 분석하여 **최대 12개 주제**의 보고서를 생성한다.

단순히 "이 회사가 잘한다"가 아니라, **"이 업계에서 이 회사가 왜 이 위치에 있는지"**를 구조적으로 보여주는 것이 목적이다.

### 계층 구조 — 관점 / 주제 / 소제목

```
관점 (13개 풀에서 선택)
  └ 주제 (최대 12개) ← 이것이 결과물의 단위
      └ 소제목 (주제 안의 하위 단위)
```

- **관점**: 13개 풀에서 고르는 카테고리 (예: "기술/R&D", "시장 전략/채널")
- **주제**: 관점을 이 기업에 맞게 구체화한 것 (예: "트랜지스터 기술 경쟁력"). **최대 12개**
- **소제목**: 주제 안에서 세부 내용을 나누는 항목 (예: "1) 원가 경쟁력", "2) 수출 경쟁력")

하나의 관점에서 여러 주제가 나올 수 있다. 예: "시장 전략/채널" 관점에서 3개 주제 + 다른 관점들에서 9개 주제 = 12개.
검색은 **주제 단위**로 하고, 품질 기준은 주제당(5개↑) + 소제목당(2개↑) 2단계로 적용한다.

---

## 입력

- **한화에어로스페이스** 
- **방산업** 
- 업계 이슈 자료는 ./reports/{업계명}_업계이슈.docx 에서 자동으로 읽는다 (있을 경우)
- **업계 이슈 자료가 없어도 실행 가능** — 검색으로 보충한다
- 사용자가 별도로 관점·주제를 지정하면 그것을 우선 반영한다

## 출력

- `{기업명}_경쟁력관점.docx`
- 경쟁력 주제 최대 12개 (하나의 관점에서 여러 주제 가능, 각 주제 안에 소제목)

## ★★★ 4대 원칙 (최우선 — 모든 규칙보다 위) ★★★

### 원칙 1: 할루시네이션 금지 — 모든 문장은 [출처] 또는 [생각]이다

**이 원칙이 이 스킬의 존재 이유다. 나머지 원칙 2~4보다 이 원칙이 위다.**

**모든 문장은 반드시 둘 중 하나로 끝난다:**
- 기사에서 가져온 팩트·수치·정보 → `문장. [매체명](URL)`
- 필자의 인사이트·해석·도입구 → `문장. [생각]`
- **[출처]도 [생각]도 없는 빈 문장은 존재해서는 안 된다**

**출처 규칙:**
- 같은 기사에서 연속으로 가져와도 **매 문장마다** 출처 표기 — 절대 생략 금지
- 출처를 못 찾으면 그 문장을 삭제한다
- URL은 web_search/web_fetch로 확보한 **실제 https:// URL**만 사용
- `[매체명]` 자체가 클릭 가능한 하이퍼링크 (URL 텍스트는 노출하지 않음)

**❌ vs ✅:**
```
❌ 잘못된 예 (출처도 [생각]도 없는 빈 문장):
한화에어로스페이스의 수출 경쟁력이 강화되고 있어요.
K9 자주포가 글로벌 시장에서 인정받고 있습니다.
→ 두 문장 모두 [출처]도 [생각]도 없음

✅ 올바른 예 (모든 문장이 [출처] 또는 [생각]):
한화에어로스페이스는 2024년 방산 수출 수주 잔고가 약 20조원을 돌파했어요. [한경](URL)
K9 자주포는 폴란드에만 672문이 수출 계약되었습니다. [한경](URL)
단순 무기 판매가 아니라 현지 생산·기술이전까지 포함한 패키지 수출이 핵심이에요. [생각]
→ 팩트 문장은 [출처], 인사이트 문장은 [생각]

✅ 숫자가 없어도 올바른 [출처] 문장:
한화에어로스페이스는 폴란드 현지에 합작법인을 설립하고 K9 자주포의 현지 생산 체제를 구축하고 있어요. [한경](URL)
방산 수출에서 기술이전과 현지 생산을 동시에 제공하는 패키지 모델이 경쟁력의 핵심으로 평가받고 있어요. [한경](URL)
→ 기사에 숫자가 없으면 숫자 없이 쓰는 것이 정확한 글이다. 숫자를 만들어내는 것이 이 스킬의 최악의 실패다.
```

### 원칙 2: 기사 원문 그대로 — 변형·창작 금지

**★ 원칙 1과 원칙 2가 충돌하면 원칙 1이 이긴다. 출처가 불확실한 정보는 넣지 않는다. 정보 없는 정확한 글 > 정보 있는 부정확한 글. 예외 없음.**

- 기사에 있는 표현(숫자, 단위, 수식어, 고유명사, 정책명)을 그대로 쓴다
- 절대 금지: 계산(더하기·빼기·곱하기·나누기), 환산, 단위 변환(억↔조, 원↔달러), 여러 기사의 정보를 조합해 새로운 정보 만들기
- "약", "최대", "이상" 등 수식어도 원문 그대로 유지
- 기사에 없는 숫자·팩트를 쓰고 싶으면 → 쓰지 않는다. 숫자 없이 맥락을 설명하는 문장이 정확한 글이다

### 원칙 3: 의미는 보존 — 바꾸면 왜곡

- 수치, 통계, 고유명사, 정책명, 사업명은 절대 바꾸지 않는다
- 서술어의 핵심 의미 (시점, 확정성, 강도, 방향, 조건 등)도 바꾸지 않는다
- 범위/조건 표현 ("최대", "일부", "~에 한해", "~를 전제로" 등)도 바꾸지 않는다
- 판단 기준: "이 표현을 바꾸면 의미가 조금이라도 달라지는가?" → 바꾸지 않는다
- **기사 발행일과 현재 시점을 비교한다.** '시행 예정'으로 쓰여 있어도 발행일이 오래됐으면 현재는 이미 시행 중일 수 있다. 최신 기사로 교차 확인이 안 되면 기사 원문 표현을 그대로 쓰고 `(YYYY년 MM월 기준)` 형태로 발행일을 명시한다

### 원칙 4: 구조만 바꾼다 — 복붙 방지

- 바꿔도 되는 것: 문장 구조, 어순 재배치, 종결어미/말투, 문장 분리/결합
- 바꾸면 안 되는 것: 원칙 2, 3에 해당하는 모든 것

## 검색 규칙

### ★ 기업 경계 규칙 — 분석 대상 기업만 다룬다 ★

- 분석 대상은 입력된 기업 하나다. 같은 그룹 계열사, 지주사, 자회사는 별개 기업이다
- 예: "신한은행"이 입력이면 → 신한투자증권, 신한자산운용, 신한카드, 신한금융지주는 분석 대상이 아니다
- 예: "한화에어로스페이스"가 입력이면 → 한화시스템, 한화오션의 독자 사업은 분석 대상이 아니다 (단, 연결 자회사로서 실적에 포함되는 경우는 그 맥락에서만 언급)

**적용 시점:**
1. **검색 시**: 검색 결과에서 계열사 단독 기사는 제외한다
2. **web_fetch 후**: 기사 본문에 계열사 내용이 섞여 있으면, 분석 대상 기업 관련 내용만 추출한다
3. **집필 시**: 계열사의 경쟁력을 이 기업의 경쟁력으로 서술하지 않는다

**예외**: 계열사 간 시너지가 이 기업의 경쟁력에 직접 영향을 주는 경우 → [생각]으로 맥락만 언급

### 검색 범위

- 2025~2026년 발행·업데이트된 기사, 자료만 활용. 가급적 최신 자료 사용
- 한국 자료만 (해외 자료 제외)

### 허용 소스

- 산업분석 리포트, 신용평가사 업종분석(한국기업평가, 나이스신용평가, 한국신용평가)
- 금융연구원·산업연구원 보고서, 협회 자료, 정부 보고서, 나무위키
- 뉴스 기사, 전문지, PDF 보고서

### 금지 소스

- 채용 사이트(캐치, 사람인, 잡코리아 등) 절대 금지
- 중앙일보 절대 금지

### 검색 금지 주제

- 주가, 주식, 주주, 배당, 시가총액, 주당순이익 등 주식·주주 관련 키워드로 검색하지 않는다
- 검색 결과에 주가/주식/주주 관련 기사가 나오면 해당 기사는 수집하지 않는다
- 보고서에 주가·주식·주주 관련 내용을 포함하지 않는다

### 검색 방법

- 원출처 우선. 기사는 원출처를 풀어서 설명하거나 맥락을 더할 때 활용
- 최신 자료 우선 (검색어에 연도 포함)
- web_search로 찾고, 반드시 web_fetch로 본문을 읽어서 수치·사실 확보
- web_search 스니펫만으로 쓴 내용 절대 금지 — 반드시 web_fetch 후 작성
- 같은 내용이 여러 기사에 있으면 대표 자료 하나만 출처로 단다
- 본문 텍스트에 있는 내용만 사용. 도표/그래프에서 직접 읽어낸 수치는 사용 금지
- 자료 발행일 확인 → "작년/올해" 등 상대 표현은 발행일 기준으로 변환

### ★ 검색 품질 기준 ★

검색은 **주제 단위**로 한다. 주제를 충분히 검색한 뒤 소제목으로 배분한다.

**주제당:**
- **바닥**: 유효 기사 최소 5개 확보
- **천장**: web_fetch 최대 10개 (과부하 방지)
- 5개 미만이면 키워드를 바꿔서 추가 검색
- 5개 이상이어도 수치·맥락이 부족하면 추가 검색 (10개 이내)

**소제목당:**
- **바닥**: 유효 기사 최소 2개
- 유효 기사가 1개뿐인 소제목은 단독으로 세우지 않는다 — 다른 소제목에 통합하거나 추가 검색

- **"유효 기사"** = 서로 다른 내용·관점을 담고 있는 기사. 같은 내용의 다른 매체 보도는 1개로 침
- 단, 유효 기사가 바닥 기준에 미달해도 해당 주제에 대한 기사가 실제로 부족하면 있는 기사만으로 작성한다. 기사 수를 채우기 위해 관련성이 낮은 기사를 끌어오지 않는다
- web_search → web_fetch로 본문 확인 → 중복이면 카운트하지 않음

### ★ 서사 설정 금지 — 기사가 이야기를 만든다 ★

**네가 이야기를 만들고 기사를 끼워 맞추는 게 아니다. 기사가 이야기를 만든다.**

```
❌ 잘못된 순서:
쓰고 싶은 결론/이야기를 먼저 설정 → 거기에 맞는 기사만 검색 → 끼워 맞추기

✅ 올바른 순서:
키워드로 검색 → 기사들을 충분히 읽음 →
기사에서 자연스럽게 보이는 흐름·패턴 발견 →
그 흐름으로 주제를 묶고 구조를 잡음 → 작성
```

- 검색 결과가 예상과 다르면 **예상을 수정한다**. 기사를 골라서 예상에 맞추지 않는다
- 특정 방향의 기사만 골라서 한쪽으로 치우친 서술을 하지 않는다
- 기사 내용에 대한 **해석과 맥락 연결은 적극 권장** — 단, 해석이 기사 내용을 왜곡하면 안 된다

## ★★★ 리서치 프로세스 ★★★

### STEP 0: 업계 이슈 읽기 + 핵심 추출 + 중복 방지 맵 작성

**./reports/ 폴더에서 업계 이슈 결과물이 있으면 읽는다:**

- `{업계명}_업계이슈.docx` 또는 `{업계명}_업계이슈.md`
- 있으면 읽고 활용한다
- **없으면 없는 대로 진행한다** — 아래 항목을 검색으로 직접 파악한다
- 사용자가 별도로 관점·주제를 지정한 경우, 그것을 우선 반영하고 나머지를 검색으로 보충한다

읽은 내용에서 아래를 내부적으로 정리한다:

1. **현재 이슈 중 차별화 지점**: 업계 이슈 중 기업 간 전략이 갈리는 지점
2. **"잘하는 회사 vs 못하는 회사"를 가르는 기준**: 이 업계에서 경쟁력의 실체

아래 항목은 2장 이슈에서 파악이 부족할 수 있다. **부족한 부분은 검색으로 보충한다:**

3. **수익구조 핵심**: 이 업계에서 돈이 어디서 나오는가, 비용이 어디에 쏠리는가
4. **경쟁구도 특성**: 과점/분산, 대형사 vs 특화사, 최근 재편 움직임
5. **규제/정책 영향**: 기업 전략에 직접 영향 주는 규제·정책 요소

#### ★★★ 중복 방지 맵 작성 (업계 이슈 자료가 있을 때 필수) ★★★

업계 이슈 자료를 읽은 뒤, **내부적으로 "중복 방지 맵"을 반드시 작성한다:**

| 업계 이슈에서 다룬 주제 | 핵심 내용 요약 (한 줄) | 이 보고서에서의 활용 방식 |
|-|-|-|
| (예: 폴란드 수출 확대) | 업계 이슈에서 K9 수출 현황·규모를 다룸 | → 이 기업의 수출 전략·현지화 모델의 **구조적 차별점**으로 심화 |

**활용 방식은 아래 3가지 중 하나만 허용된다:**

1. **심화**: 업계 이슈에서 다룬 현상을 **이 기업 관점에서 "왜 이 기업이 여기서 강한가"로 한 단계 깊게** 분석. 업계 이슈가 "무엇이 일어나고 있는가"였다면, 여기서는 "이 기업이 왜/어떻게 그 흐름에서 경쟁력을 가지는가"를 쓴다
2. **전환**: 업계 이슈에서 다룬 주제의 완전히 다른 측면을 다룬다 (예: 업계 이슈가 "규제 변화"를 다뤘으면, 여기서는 "그 규제에 대한 이 기업의 선제 대응 전략"을 다룬다)
3. **스킵**: 업계 이슈에서 충분히 다뤘고 기업 경쟁력 관점에서 심화할 여지가 없으면 이 보고서에서 다루지 않는다

**절대 금지:**
- 업계 이슈에서 쓴 내용(팩트, 수치, 기사 해석)을 그대로 또는 말만 바꿔서 반복하는 것
- 업계 이슈와 같은 기사를 인용하면서 같은 내용을 전달하는 것
- 업계 이슈에서 "업계 전체 동향"으로 다룬 내용을 주어만 "이 기업은"으로 바꿔서 쓰는 것

### ★★★ 업계 이슈 보고서와의 차별화 — 이 보고서의 존재 이유 ★★★

**이 보고서는 업계 이슈 보고서의 연장선이 아니라, 완전히 다른 질문에 답하는 글이다.**

- **업계 이슈 보고서의 질문**: "이 업계에서 지금 무슨 일이 벌어지고 있는가?"

**집필 시 모든 소제목에서 아래를 자기 검증한다:**

1. **"이 내용이 업계 이슈 보고서에 이미 있지 않은가?"** → 있다면 그대로 쓰면 안 된다
2. **"업계 이슈 보고서와 같은 주제라면, 한 단계 더 깊이 들어갔는가?"** → 업계 이슈가 "현상"을 다뤘다면, 여기서는 "이 기업의 전략·구조·역량" 관점에서 심화해야 한다
3. **"업계 이슈 보고서와 같은 기사를 인용하는가?"** → 같은 기사를 써도 되지만, 같은 내용을 전달하면 안 된다. 그 기사에서 업계 이슈 보고서가 다루지 않은 기업 고유의 팩트·수치를 뽑아야 한다

**심화 분석의 구체적 방향:**

- 업계 이슈가 "시장이 커지고 있다"를 다뤘으면 → 이 기업이 **그 성장에서 어떤 포지션을 잡고 있는지, 경쟁사 대비 어떤 차별점이 있는지**

## ★★★ 문서 생성 방식 ★★★

### ★ 출처 정확성 규칙 (JS 코드 작성 시 필수) ★

JS 코드로 docx를 생성할 때, **문장과 출처가 뒤섞이는 것을 방지**하기 위해 아래 순서를 반드시 따른다:

1. **전체 흐름 설계**: 주제 순서, 소제목 간 연결, 핵심 메시지를 먼저 잡는다 (깊이·맥락 확보)
2. **소제목 단위로 JS 코드 작성**: 하나의 소제목에 들어갈 문장들과 출처를 먼저 완성한 뒤, 다음 소제목으로 넘어간다. 한번에 전체를 작성하지 않는다
3. **★ URL 상수 선언 (필수)**: 소제목 JS 코드 시작 전, 해당 소제목에서 인용하는 기사 URL을 **매체명_키워드** 형태의 상수로 선언한다. 문장에서 hyperlink를 넣을 때 URL 문자열을 직접 쓰지 말고 반드시 상수명을 참조한다. 상수명이 곧 어떤 기사인지 알려주므로, 문장 내용과 출처가 맞는지 코드만 봐도 바로 확인된다.

```js
// ★ 소제목 시작 전: 이 소제목에서 쓸 기사 URL 상수 선언
const 한경_GA수수료상한제 = "https://...실제URL...";
const 매경_설계사이탈 = "https://...실제URL...";

// ★ 문장 작성 시: URL 문자열 직접 복붙 금지, 반드시 상수명 사용
new TextRun({ text: "금융위원회는 2025년 3월부터 GA 수수료 상한제를 시행했어요. " }),
new ExternalHyperlink({ link: 한경_GA수수료상한제, children: [...] }),
//                            ↑ 변수명만 봐도 이 문장과 맞는지 바로 확인 가능
```

4. **문장-출처 짝 검증**: 각 문장 뒤의 hyperlink가 참조하는 **상수명**이 그 문장의 내용과 일치하는지 확인한다. 상수명과 문장 내용이 어긋나면 즉시 수정한다

## ★★★ 출력 전 체크 (출력에 포함 금지) ★★★

### 출처 (가장 중요):
- **모든** 문장이 `[매체명](URL)` 또는 `[생각]` 둘 중 하나로 끝나는가? [출처]도 [생각]도 없는 빈 문장이 하나라도 있으면 수정
- 같은 출처에서 연속 문장이 나올 때도 매 문장마다 출처가 반복 표기되어 있는가
- 매체명만 텍스트로 적은 곳이 없는가 (절대 금지 — 반드시 하이퍼링크)
- 하단에 인용 기사 **전부** 나열했는가? 빠진 출처가 없는가
- 출처 URL이 실제 web_fetch로 접근한 URL만 사용했는가

### 정확성:
- [출처] 문장의 모든 정보(숫자, 고유명사, 시점, 진행상태)가 web_fetch로 읽은 기사 원문에 실제로 존재하는가 — 존재하지 않는 정보가 있으면 해당 정보를 삭제한다
- 기사 원문의 표현을 계산·환산·조합하지 않았는가
- 법령·정책의 진행 상태(논의/발의/시행 등)를 기사 원문대로 구분했는가

### 내용:
- ESG, 주주가치 제고, 지속가능경영 등 범용 주제가 섞이지 않았는가
- 최소 3개 이상의 업종·기업 고유 관점이 포함되어 있는가
- 기업의 전략/미래 비전 관점이 포함되어 있는가
- web_search 스니펫만으로 쓴 내용이 없는가

### ★ 업계 이슈 보고서와의 중복 체크 (업계 이슈 자료가 있을 때 필수):
- 업계 이슈 보고서에서 다룬 내용을 그대로 또는 말만 바꿔서 반복한 소제목이 없는가
- 업계 이슈 보고서와 같은 주제를 다루는 소제목이 있다면, "이 기업은 왜/어떻게" 관점에서 한 단계 심화되었는가
- 업계 이슈 보고서의 "업계 전체 동향" 서술에서 주어만 "이 기업은"으로 바꾼 문장이 없는가
- 중복 방지 맵에서 정한 활용 방식(심화/전환/스킵)이 실제 집필에 반영되었는가

### 분량:
- 기사 내용을 2~3줄로 압축하지 않았는가? 충분히 풀어서 썼는가
- [생각]이 충분한가? 기사 4~6문장 → [생각] 3~5문장 비율이 지켜지는가

### 문체:
- 친절하게 옆에서 설명하는 존댓말 톤이 유지되고 있는가
- 소제목 하나당 최소 10문장 이상인가 (5~6문장으로 끝나지 않았는가)

## ★★★ 자동화 환경 실행 규칙 (Routine 전용) ★★★

이 섹션은 자동화 환경(Claude Code Routine 등)에서 이 스킬이 실행될 때 적용되는 규칙이다. 사용자가 직접 호출하는 일반 환경에서는 기존 규칙대로 작동한다.

### 인풋 자동 읽기

자동화 환경에서는 사용자에게 묻지 않고 다음 경로의 파일을 자동으로 읽는다:

1. **기업명**: `inputs/기업명.txt` — 한 줄, UTF-8. 예: `산은캐피탈`
2. **업계명**: `inputs/업계명.txt` — 한 줄, UTF-8. 예: `캐피탈업`
3. **업계 특성 보고서 (필수 참고)**: `output/{업계명}_업계특성.docx`
   - 1번 스킬에서 직전에 생성된 파일
   - 이 보고서를 처음부터 끝까지 통독한 뒤 작업 시작

**필수 파일 누락 시:** `inputs/기업명.txt`, `inputs/업계명.txt`, `output/{업계명}_업계특성.docx` 중 하나라도 없으면 즉시 중단하고 어떤 파일이 없는지 출력한다.

### ★ 1번 결과물과 중복 회피 (CRITICAL) ★

`output/{업계명}_업계특성.docx`를 먼저 통독하고, 거기서 이미 다룬 내용은 **이 보고서에서 중복 서술하지 않는다.**

**구체 절차:**
1. 1번 보고서의 모든 소제목과 핵심 논지를 리스트업한다 (내부 메모용)
2. 경쟁력 관점·주제를 도출할 때, 1번에서 이미 일반론으로 다룬 업계 구조·특성은 **간단히 참조하고 넘어간다**:
   - ❌ 1번에서 다룬 "방산업의 정부 의존도" 구조를 다시 길게 풀이
   - ✅ "1번 보고서에서 다룬 정부 의존 구조 위에서, 한화에어로스페이스가 어떻게 차별화하는가"로 빠르게 전환
3. 이 보고서는 **기업 특수성**에 집중한다. 업계 일반론은 1번 보고서가 담당한다.

**중복 판정 기준:**
- 같은 통계·수치·트렌드를 거의 같은 방식으로 서술 → 중복
- 같은 사실을 다른 각도(기업 전략·경쟁력 관점)로 재해석 → 중복 아님 (허용)

### 아웃풋 저장 위치

- 경로: `output/{기업명}_경쟁력관점.docx`
- 예: 인풋이 `산은캐피탈`이면 → `output/산은캐피탈_경쟁력관점.docx`

**파일명 규칙 (절대 위반 금지):**
- 폴더명 정확히 `output`
- 공백·특수문자 없음 (`_` 외 사용 금지)
- 확장자 정확히 `.docx`
- 기업명은 `inputs/기업명.txt`에 적힌 그대로 사용

이 파일은 다음 단계인 3번(섹션 보강)과 4번(팩트체크)이 인풋으로 받아간다. 이름이 한 글자라도 다르면 자동화 체인이 끊긴다.
